# Sesion 2026-03-31 — Logger class + Haptics 30/30

## Objetivo

Diagnosticar por que haptics (dataset 05) fallaba con OOM kills en el cluster,
e implementar logging robusto para entender el comportamiento del algoritmo.

## Cambios realizados

### Nuevos archivos
- `logger.h` / `logger.cpp` — Logger class estilo Python
  - Niveles: DEBUG, INFO, WARNING, ERROR
  - Timestamps: `[sec.usec]` via `gettimeofday`
  - Macros: `LOG_INFO(msg)`, `LOG_DEBUG(msg)`, `LOG_WARNING(msg)`, `LOG_ERROR(msg)`
  - Global: `extern Logger g_logger`

### Archivos modificados
- `main.cc` — 6to arg `results_dir`, creacion automatica de subdirectorios,
  tracking de RAM via `/proc/self/status` VmRSS, reemplazo de `logFile` por `g_logger`
- `bridge.cpp` — Timing de MCFP (build/solve/extract), stagnation detection
  (1K/5K/10K evals sin mejora), logging de NEW BEST con detalles por cluster
- `diversity.cc` — Log de 200 individuos iniciales, stats por generacion
  (best/mean/std/D) en cada generacion
- `solution.cpp` — Usa Logger
- `de.h` — Incluye logger.h en lugar de extern logFile
- `Makefile` — Agrega logger.o, `-march=x86-64` en lugar de `-march=native`

## Problemas encontrados y resueltos

| Problema | Causa | Solucion |
|----------|-------|----------|
| "Illegal instruction" en cluster | `-march=native` compila para CPU del login node | Cambiar a `-march=x86-64` |
| OOM kills con ntasks=6 (14/30) | 6 procesos haptics (5460 vars) exceden RAM del nodo | Usar ntasks=2 (secuencial) |
| ntasks=1 falla en 2s | MPI_Scheduler necesita min 2 procesos | Usar ntasks=2 |
| result.txt vacios | OOM kill antes de escribir resultado | Reducir concurrencia |

### Analisis de convergencia (seed 1)
- El algoritmo NO converge — sigue mejorando hasta eval 95929/100000
- Burst tardio en evals 91K-95K (mecanismo L-SHADE de reduccion de poblacion)
- Estancamiento mas largo: ~39K evals (52201 a 91684)
- Conclusion: 100K evals es insuficiente para 5460 dimensiones

### Causa del gap con BKS
No es un bug. Es maldicion de la dimensionalidad:
- DE con 200 individuos en espacio de 5460D no puede explorar adecuadamente
- MCFP hace su parte correctamente (asignacion optima dado los centros)
- El problema es que DE no encuentra buenos centros en 5460D
